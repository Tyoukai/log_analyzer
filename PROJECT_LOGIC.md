# 大掌柜日志分析服务 - 项目核心逻辑说明文档

本文档旨在帮助后续维护者快速理解该 Python 日志分析项目的架构定位、数据流转逻辑及引擎核心实现，降低上手与维护成本。

## 1. 架构定位

本服务是一个**无数据库依赖的轻量级日志模板训练与匹配引擎**。它的唯一职责是为 PM (Process-Management) 项目提供纯粹的算法分析能力。

### 1.1 与 PM 项目的边界

- **Python 脚本（本服务）**：负责训练模板、内存匹配、维护本地模板库状态文件。不碰数据库、不发告警。
- **PM 项目**：负责消费日志、调用本服务 API 获取 `template_id`，随后负责将模板信息写入 MySQL/TDengine 并管理模板生命周期及告警。

### 1.2 运行模式演进

本项目的生命周期分为两个阶段，由 `config.py` 中的 `enable_consumer` 开关控制：

- **冷启动与积攒阶段（开关开启）**：
  服务启动后台线程，直接订阅大掌柜 Kafka 历史日志流，进行大批量的离线/在线混合训练。
- **平稳运行阶段（开关关闭）**：
  历史数据训练饱和后，运维人员修改配置关闭 Kafka 订阅开关并重启。此后，引擎变为纯被动模式，完全依赖 PM 项目实时调用 API（增量查询）来维持模板库的持续进化。

## 2. 核心训练与处理逻辑

项目的核心逻辑收敛在 `log_template_engine.py` 文件中，底层依赖于 `Drain3` 聚类引擎。完整的单条日志处理流如下：

### 2.1 阶段一：严格格式校验与重组 (`preprocess_log`)

接收到的原始数据是 FileBeat 采集的 JSON 字符串。
1. **强校验**：如果缺少核心要素（`log_level`, `loginfo`, `source_ip`, `timestamp`, `log.file.path`），会直接抛出 `ValueError`。
2. **级别过滤**：系统目前强制只处理 `ERROR` 级别的日志，其余级别直接舍弃返回 `None`。
3. **内容重组**：将纷繁复杂的 JSON 丢弃，提取 5 个要素拼接成统一的高质量纯文本，防止 JSON Key 排序或无关字段干扰聚类算法：
   ```text
   [时间戳] [日志级别] [IP地址] [文件路径] 真实的loginfo日志内容
   ```

### 2.2 阶段二：正则表达式掩码预清洗

大掌柜系统服务众多且老旧，单纯扔进聚类引擎会导致“模板爆炸”。引擎在重组后，会运用 `TemplateMinerConfig.masking_instructions` 执行高强度的正则掩码处理（见代码中的 `RegexMaskingInstruction`）：
- 剥离杂乱 JSON Payload -> `<JSON>`
- 将行首重组信息替换 -> `<TIMESTAMP>`, `<LEVEL>`, `<IP>`, `<PATH>`
- 将日志内部的线程号、Java类及行号替换 -> `<THREAD>`, `<CODE_LINE>`
- UUID 和所有散落数字替换 -> `<UUID>`, `<NUM>`

### 2.3 阶段三：解析树聚类与持久化

1. 引擎将掩码后的文本送入 Drain3 的固定深度解析树。
2. 若匹配成功，直接返回模板的 MD5 (`template_id`)。
3. 若未命中，引擎**即时在内存树中开辟新节点**。
4. **自动触发本地持久化**：Drain3 配置的 `FilePersistenceHandler` 会捕捉到树的变更，自动将最新的树状态序列化回本地磁盘文件 `drain3_state.bin`。

## 3. 接口调用与异常应对机制

项目的对外交互接口在 `api.py` 中。

### 3.1 正常响应闭环
PM 调用 `POST /api/match_log_template` -> 引擎走完上述训练逻辑 -> 返回 `template_id` 和 `is_new` 标识。如果是新模板（`is_new: true`），此时硬盘上的 `.bin` 文件也已经同步更新完毕。

### 3.2 异常拦截机制
由于引擎对日志进行了强校验，以下情况会被抛出 `ValueError`：
- 不是有效的 JSON。
- 缺少核心要素。

在系统各处，对 `ValueError` 的处理策略不同：
- **API 端**：捕获到后，立刻转化为 HTTP 400 状态码返回给 PM，提示 PM 其上报或传递的格式存在漏洞。
- **Kafka 消费端**：捕获到后，仅仅在控制台 `pass`（丢弃），不中断后台数十万条消息的消费循环进程。

### 3.3 服务重启与状态恢复
每次服务重启，都会在 `LogTemplateEngine.__init__` 时优先调用 `FilePersistenceHandler` 去同级目录下寻找 `drain3_state.bin`。
如果存在，直接反序列化恢复解析树内存。因此服务具有极高的稳定性，不用担心进程意外退出导致的训练成果清零。
